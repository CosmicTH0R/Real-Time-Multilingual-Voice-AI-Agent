# Latency Optimization & Measurements

Our Voice AI agent aims for an absolute end-to-end latency (Speech end to first audio byte heard) of **< 450ms**. 

## Pipeline Parallelism Breakdown

We log strict Unix timestamps throughout the lifecycle of an utterance (`LatencyTracker`). The typical breakdown achievable with this architecture:

| Component | Operation | Target Duration | Output |
|-----------|-----------|-----------------|--------|
| **STT** (Deepgram) | Streaming VAD silence detection + Final Transcript generation | `< 100ms` | Text |
| **Memory** (Redis/PG) | `MemoryManager` fetch. Fired in parallel with STT finalization phase. | `< 20ms` | Context |
| **Agent Core** (LLM) | Streaming API call to OpenAI / Gemini. Time-to-First-Token (TTFT). | `< 200ms` | Stream |
| **TTS** (Google / 11L)| Chunked synthesis. Dispatches the first clause over WebSocket. | `< 100ms` | Audio |
| **Network** | WebSocket overhead and serialization. | `< 30ms` | Frame |

## Key Strategies Implemented

1. **VAD Chunking vs Speech-End Detection**: Instead of waiting for the user to completely stop talking and manually hitting a "send" button, Deepgram Endpointing dictates the utterance boundary to slice latency.
2. **Chunked TTS Streaming**: We do not wait for the LLM to complete its entire massive response before we send it to TTS. As soon as a logical sentence break (`.`, `?`, `!`, `\n`) is generated by the LLM, that single clause is sent to TTS. TTS generates audio for *just* that clause and streams the byte array to the client while the LLM continues generating the subsequent clauses in the background.
3. **Optimized DB Connections**: Using asynchronous SQLAlchemy connection pooling with `asyncpg` combined with Redis caching guarantees sub-millisecond local reads for memory hydration.
